# Khetify Barcode Architecture

Three barcode types span the platform. All are generated by us and are
self-describing (a scan tells the system exactly what was scanned).

## 1. Lot barcode (Code 128)
```
K-L-<companyShort>-<sku>-<lot>
e.g.  K-L-AGR-FRT102-2606001
```
- `companyShort` ‚Äî up to 4 alphanumerics identifying the tenant
- `sku` ‚Äî product SKU
- `lot` ‚Äî the lot / batch number
- Printed on the lot label (one per lot). Renders as Code 128 (denser than
  the legacy Code 39 used for short codes).

## 2. Unit (product) serial barcode
```
<LOTKEY>-<seq>
e.g.  ABSAMIO012-001
```
- `LOTKEY` ‚Äî the lot / batch number, uppercased with non-alphanumerics stripped
- `seq` ‚Äî a per-lot sequence number, zero-padded to 3 digits, allocated from an
  atomic Counter block so serials never collide within a lot
- There is NO prefix. Serials used to carry a `K-U-` prefix; it was dropped, but
  `resolveScan` still tolerates a legacy `K-U-` on labels printed before the
  change (and `scripts/stripUnitPrefix.js` migrates existing DB serials).
- One per physical unit. Uniqueness is enforced by a unique index on
  `UnitSerial.serial`; cross-lot collisions are astronomically unlikely given
  the 7-char lot segment, and the index is the hard backstop.
- A QR payload is also stored: `{"t":"unit","s":"<serial>"}` for camera scanning.

## 3. Location barcode
```
<warehousePrefix>-<zone>-<rack>-<shelf>-<bin>
e.g.  WH1-A-R03-S2-B07
```
- This is `Location.fullCode` from the storage hierarchy (Sprint 1). Printed on
  every bin label.

## Parent ‚Üí child relationship
```
Product (SKU, trackSerial flag)
  ‚îî‚îÄ Lot  (Inventory row: product + warehouse + batch)        ‚Üê lot barcode
       ‚îî‚îÄ Unit (UnitSerial)  √ó N                              ‚Üê unit serials
            ‚îî‚îÄ UnitEvent √ó M   (append-only movement history)
```
One lot contains many units; every unit carries its lot/batch, so a unit scan
resolves the full chain Product ‚Üí Lot ‚Üí Unit, and the unit's `orderId`/
`customerId` close the loop to the buyer for recall.

## Generation workflow
1. Stock is received (GRN) ‚Üí an Inventory (lot) row exists.
2. `POST /api/units/generate { inventoryId, qty }` reserves a counter block and
   bulk-inserts `qty` serials with status `generated`; the product is flagged
   `trackSerial`.
3. Labels are printed (status ‚Üí `printed`).

## Scan workflow
`POST /api/scan { code }` ‚Üí `resolveScan` resolves in order: `K-L-` ‚Üí lot, then
an exact unit-serial match (prefix-less; a legacy `K-U-` is stripped first),
then a known `fullCode` ‚Üí location, otherwise a raw lot/batch number. It returns
the entity plus role-aware **next actions**
(putaway / pick / pack / dispatch ‚Ä¶). USB scanners act as keyboard wedges; the
frontend Scan box reads the rapid keystrokes and posts the code.

```
generated ‚Üí printed ‚Üí in_stock ‚Üí picked ‚Üí packed ‚Üí shipped ‚Üí sold
                                   ‚Üë                              ‚Üì

```
Transitions go only through `barcodeService.transitionUnits`, which validates
the move and writes a `UnitEvent`. Workflow services (putaway, pick, pack,
dispatch, POD, returns) call these ‚Äî see Sprints 4‚Äì5.

## Recall
`POST /api/recall { lotNumber }` (capability `recall:execute`, company_admin
only): marks every not-yet-sold serial `recalled` (blocking it from picking)
and returns the full distribution ‚Äî warehouses/bins still holding stock + every
order/customer that received a unit from the lot. The action is written to the
AuditLog.
